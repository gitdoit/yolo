# VLM 推理性能测试报告

> 目的：评估不同视觉语言模型在 CPU / GPU 环境下的推理性能，用于判断部署方案辅助 YOLO 做洪水识别。

## 测试环境

| 项目 | 配置 |
|------|------|
| 推理框架 | Ollama (本地部署) |
| 图片数量 | 20 张 (flood 数据集 split_train) |
| 图片缩放 | 长边 ≤ 640 (JPEG quality=85) |
| 输出限制 | `num_predict: 256` |
| 测试脚本 | `scripts/19_vlm_batch_benchmark.py` |

## 详细分析

### CPU 测试

### qwen3.5:0.8b — 最快，推荐用于 CPU 部署

- **运行记录**: `run_20260403_181538`
- 平均 5s/张，延迟极稳定 (4.66s ~ 5.74s)
- 内存：~34.5 MB (Python 进程)
- `think: false` 生效，无 thinking 开销
- 3 张误判为非洪水 (准确率 85%)
- prompt tokens 统一 ~305 (640 缩放后)

### qwen2.5vl:3b — 最准，但速度中等

- **运行记录**: `run_20260403_174035`
- 该轮测试未缩放图片，大图 prompt tokens 高达 2726，小图仅 689
- 大小图延迟差异巨大 (1s ~ 27.9s)
- 19/20 识别正确 (准确率 95%)，唯一 dirty 图正确判否
- 输出含中文 reason，可解释性好

### moondream:latest — 指令遵循差，不推荐

- **运行记录**: `run_20260407_100119`
- 模型大小 1.9B，非中文原生模型
- 平均 14.52s/张，正常响应 ~13s，幻觉输出撑满 token 上限后达 22-25s
- 内存：~34.2 MB (Python 进程)
- **55% 解析失败 (11/20)** — 中文 prompt 指令遵循极差
- 常见问题：生成 a-z 单字母键的超长 JSON、JSON 数组等幻觉输出
- 0 张返回 False — 未展现真正判断能力，全部倾向输出 True
- 不支持结构化 JSON 输出，不适合需要精确响应格式的场景

### qwen3-vl:2b — 最慢，thinking 无法关闭

- **运行记录**: `run_20260407_094618`
- Ollama `think: false` 参数对此模型不生效
- thinking 消耗大量 token，4 张图因 256 token 上限被截断，JSON 输出不完整
- 即使调大 `num_predict`，thinking 仍浪费推理时间
- prompt tokens 统一 305 (640 缩放)，但 eval 仍慢

### GPU 测试

### qwen3.5:0.8b (GPU) — 延迟极低，准确率提升

- **运行记录**: `run_20260407_101200`
- 平均 0.59s/张，延迟极稳定 (0.52s ~ 0.83s)
- CPU → GPU 加速 **8.6 倍** (5.06s → 0.59s)
- 准确率从 CPU 的 17T/3F 提升到 19T/1F (仅 dirty 正确判否)
- GPU 下 0.8B 模型已可用于实时视频流逐帧验证

### qwen2.5vl:3b (GPU) — 准确率不变，速度提升明显

- **运行记录**: `run_20260407_101254`
- 平均 2.13s/张，稳态约 1.8s (首张 7.77s 含模型加载 6.3s)
- CPU → GPU 加速 **4.9 倍** (10.48s → 2.13s)
- 准确率 19T/1F/0U，与 CPU 测试一致 (dirty 图正确判否)
- 加速比相对低，因 3B 模型本身较大，GPU 显存占用更多

### qwen3-vl:2b (GPU) — 速度大幅提升，thinking 问题不变

- **运行记录**: `run_20260407_101441`
- 平均 1.49s/张 (首张 5.32s 含加载)，稳态 0.7s ~ 1.9s
- CPU → GPU 加速 **11.0 倍** (16.33s → 1.49s)
- 仍有 4 张解析失败 (15T/1F/4U)，thinking 在 GPU 下同样不可关闭
- GPU 加速了推理但未解决 thinking 浪费 token 的根本问题

### moondream:latest (GPU) — 速度最快，解析失败率改善

- **运行记录**: `run_20260407_101612`
- 平均 1.06s/张，稳态仅 **~0.30s/张** (首张 15.45s 含加载 2.4s + GPU 初始化)
- CPU → GPU 加速 **13.7 倍** (14.52s → 1.06s)
- 解析失败从 CPU 55% (11/20) 降到 GPU 20% (4/20)
- 仍然 0 张返回 False，全部倾向输出 True
- GPU 下推理速度极快 (P50=0.30s)，但指令遵循问题仍在

### openbmb/minicpm-v4 (CPU) — 极慢，准确率低，不推荐

- **运行记录**: `run_20260407_104339`
- 参数量 3.6B，OpenBMB 官方发布的多模态模型
- 平均 31.19s/张，延迟稳定 (29.03s ~ 36.52s)
- 内存：~34.3 MB (Python 进程)
- 准确率仅 **60%** (11T/9F)：19 张洪水图中仅 11 张识别正确，8 张漏检
- 1 张非洪水 (dirty) 正确判否
- 所有图片均返回有效 JSON，解析失败率 0%
- **洪水召回率 57.9% (11/19)**，远低于 qwen2.5vl (100%) 和 qwen3.5 (89%)

### openbmb/minicpm-v4 (GPU) — 无加速，仍极慢

- **运行记录**: `run_20260407_105508`
- 平均 31.87s/张 (29.52s ~ 37.63s)，与 CPU 几乎无差异
- CPU → GPU 加速 **0.98 倍** (实际略慢，无加速效果)
- 疑似原因：Ollama 对 minicpm-v4 模型架构的 GPU 推理支持不完善，或模型 GGUF 量化方式不支持 GPU offload
- 准确率与 CPU 一致 (11T/9F)
- 不推荐在任何场景使用——速度是 qwen3.5:0.8b GPU 的 **54 倍**，准确率还更低

### smolvlm2-256m (CPU) — 极快但完全不可用

- **运行记录**: `run_20260407_111905` (中文 prompt), `run_20260407_112650` (英文 prompt)
- 参数量仅 0.26B (256M)，世界上最小的多模态 VLM
- Ollama 社区模型: `ahmadwaqar/smolvlm2-256m-video:q8_0` (Q8 量化，279MB)
- **中文 prompt**: 平均 3.50s/张，85% 解析失败 (17/20)，生成随机代码/虚构 URL 等幻觉
- **英文 prompt**: 平均 2.38s/张 (更快)，但 **100% 解析失败 (20/20)**
  - 模型能用英文描述图片内容（如 "river with brown water", "flooded street"），说明视觉理解有基础能力
  - 但完全不遵循 JSON 格式指令，输出自然语言描述而非结构化 JSON
  - 部分回答前缀奇怪的时间戳 (如 "2023-06-27 14:59:38 +02:59")
- 0.26B 参数不足以同时理解图片内容 + 遵循格式指令

### smolvlm2-256m (GPU) — 无加速，同样不可用

- **运行记录**: `run_20260407_112206` (中文 prompt), `run_20260407_112914` (英文 prompt)
- **中文 prompt**: 平均 3.48s/张，85% 解析失败 (3T/0F/17U)
- **英文 prompt**: 平均 2.28s/张，100% 解析失败 (0T/0F/20U)
- CPU/GPU 无差异 (1.01x)，模型极小 CPU 已足够快
- 不推荐在任何场景使用——虽然速度快，但完全无法完成结构化洪水判断任务

## CLIP 零样本分类对比 (非 VLM 方案)

> CLIP (Contrastive Language-Image Pretraining) 不是传统 VLM，而是将图片和文字映射到同一向量空间，通过余弦相似度做零样本分类。不需要 Ollama，不需要生成式推理，直接对比相似度得分即可判断。

### 测试方案

| 项目 | 配置 |
|------|------|
| 模型 | OpenCLIP **ViT-B-32** (openai 预训练, 605MB) |
| 框架 | open-clip-torch 3.3.0 + PyTorch 2.9.1 |
| 正面文本 | 6 条英文描述 (flooded street, standing water 等) |
| 负面文本 | 6 条英文描述 (normal dry street, clear road 等) |
| 判定逻辑 | 正面/负面平均相似度之差 > 0 → 判为洪水 |
| 测试脚本 | `scripts/20_clip_flood_benchmark.py` |

### 性能对比

| 方案 | 设备 | 平均耗时 | 准确率 | 模型加载 |
|------|------|----------|--------|----------|
| **CLIP ViT-B-32** | **GPU** | **0.0495s** | **20/20 (100%)** | 100s (首次下载) / ~3s (缓存) |
| **CLIP ViT-B-32** | **CPU** | **0.0806s** | **20/20 (100%)** | 2.95s |
| qwen3.5:0.8b (VLM) | GPU | 0.59s | 19/20 (95%) | ~6s |
| qwen3.5:0.8b (VLM) | CPU | 5.06s | 17/20 (85%) | ~2s |
| qwen2.5vl:3b (VLM) | GPU | 2.13s | 19/20 (95%) | ~6s |
| qwen2.5vl:3b (VLM) | CPU | 10.48s | 19/20 (95%) | ~2s |

### CLIP vs VLM 速度对比

| 对比 | VLM 耗时 | CLIP 耗时 | CLIP 加速倍数 |
|------|----------|-----------|---------------|
| vs qwen3.5 GPU | 0.59s | 0.0495s | **11.9x** |
| vs qwen3.5 CPU | 5.06s | 0.0806s | **62.8x** |
| vs qwen2.5vl GPU | 2.13s | 0.0495s | **43.0x** |
| vs qwen2.5vl CPU | 10.48s | 0.0806s | **130.0x** |

### CLIP 详细结果

**GPU 测试** (`run_20260407_153346`):
- 20/20 全部正确：19 张洪水 → FLOOD，1 张非洪水 → NORMAL
- 平均 confidence gap: +0.0448 (洪水) / -0.0254 (非洪水)
- 推理速度极稳定：0.018s ~ 0.061s

**CPU 测试** (`run_20260407_153505`):
- 准确率与 GPU 完全一致 (20/20)
- 平均 0.08s/张，约 GPU 的 1.6 倍
- CPU 下 CLIP 仍比最快 VLM (qwen3.5 GPU) 快 **7.3 倍**

### CLIP 优势与局限

**优势：**
- 速度碾压所有 VLM — GPU 下单张 0.05s，可轻松处理实时视频流 (20+ FPS)
- CPU 下也仅 0.08s/张，远优于 VLM CPU 最快的 5s/张
- 100% 准确率（当前测试集）
- 无需 Ollama 部署，纯 PyTorch 推理
- 无解析失败问题 — 输出是数值相似度，不需要 JSON 解析
- 模型仅 605MB，远小于 VLM 模型

**局限：**
- 不提供推理说明 (reason) — 只有相似度得分，没有"为什么判为洪水"
- 分类能力取决于文本描述的质量 — 需要精心设计正面/负面 prompt
- confidence gap 绝对值较小 (0.03~0.05)，边界案例可能不稳定
- 仅做二分类，无法检测洪水区域/范围

## 结论

### CPU 部署

| 场景 | 推荐模型 |
|------|----------|
| **无 GPU 实时分类 (最快)** | **CLIP ViT-B-32** (0.08s/张，12+ FPS，100% 准确) |
| 无 GPU 服务器 + 需要理由 | **qwen3.5:0.8b** (5s/张，可接受) |
| 离线批量验证/需要理由 | **qwen2.5vl:3b** (准确率最高) |
| 不推荐 | qwen3-vl:2b (慢且 thinking 不可控) |
| 不推荐 | moondream (中文指令遵循差，55% 解析失败) |
| 不推荐 | openbmb/minicpm-v4 (31s/张极慢，召回率仅 58%) |
| 不推荐 | smolvlm2-256m (85% 解析失败，中文不可用) |

### GPU 部署

| 场景 | 推荐模型 |
|------|----------|
| **实时视频流分类 (最快)** | **CLIP ViT-B-32** (0.05s/张，20+ FPS，100% 准确) |
| 实时视频流 + 需要理由 | **qwen3.5:0.8b** (0.59s/张，可逐帧) |
| 高准确率 + 合理速度 | **qwen2.5vl:3b** (2.13s/张，19/20 正确) |
| 不推荐 | qwen3-vl:2b (thinking 仍不可控) |
| 不推荐 | moondream (解析改善但 0 False，无判断力) |
| 不推荐 | openbmb/minicpm-v4 (无GPU加速，31s/张，准确率仅60%) |
| 不推荐 | smolvlm2-256m (无GPU加速，85%解析失败) |

### 部署建议

1. **CLIP 是洪水二分类场景的最优选择** — 速度是最快 VLM 的 12 倍，准确率 100%，无需 Ollama
2. 640 缩放是必要的——消除大图导致的延迟波动 (从 1s~28s 变为 4.7s~5.7s)
3. **有 GPU**：CLIP 0.05s/张可实时逐帧；如果需要推理说明则用 qwen3.5:0.8b (0.59s/张)
4. **无 GPU**：CLIP 0.08s/张仍可实时；如果需要自然语言理由则用 qwen3.5:0.8b (5s/张)
5. **推荐组合方案**：YOLO 检测 → CLIP 快速过滤 → VLM 仅对置信度低的帧做详细分析
4. GPU 加速 5~14 倍不等，小模型受益更大 (moondream 13.7x > qwen2.5vl 4.9x)；minicpm-v4 完全无加速
5. 生产环境建议增加 YOLO 检测置信度阈值过滤，只对疑似洪水帧调用 VLM，降低总调用量
6. minicpm-v4 虽然参数量大 (3.6B)，但洪水检测准确率反而最低 (60%)，说明通用 VLM 大 ≠ 准
7. smolvlm2-256m (0.26B) 验证了极小模型的下限——速度虽快 (3.5s CPU) 但完全无法理解中文 prompt，85% 解析失败
